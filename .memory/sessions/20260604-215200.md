# Session: 2026-06-04 21:52

## Summary

诊断并修复 M4.3 `/alerts` 已处理预警回弹 bug：打开“已处理”筛选触发同步后，已处理预警不再被重新创建为待处理副本，重复已处理幽灵记录也会被清理。

## User Feedback

用户在 `/alerts` 标记一条预警为已处理后，待处理数从 7 变 6；但点击“已处理”筛选后，待处理数又回到 7，刚处理的条目重新出现在待处理，同时产生多个已处理幽灵条目。

## Diagnosis

- 复现 seam：`tests/alert-api.test.ts` 的 API helper 测试。
- 根因：`syncAlerts()` 只查询 `status=PENDING` 的已有预警。用户把仍满足规则的预警改成 `RESOLVED` 后，下一次 `GET /api/alerts` 同步认为该 item/type 没有待处理预警，于是重新创建 `PENDING`。
- 正确语义：同 item/type 已存在 `RESOLVED` 时，规则仍成立也不应重建 `PENDING`；如果历史上已经有 `RESOLVED`，同 key 的 `PENDING` bug 副本应删除；重复 `RESOLVED` 幽灵记录只保留最早一条。

## Fix

- 更新 `lib/api/alerts.ts`：同步阶段查询同 item/type 的 `PENDING` 和 `RESOLVED`。
- 创建预警时使用所有 existing keys 去重，不只看 pending keys。
- pending 收尾时，如果同 key 不再符合规则，则更新为 `RESOLVED`；如果同 key 已有 resolved 记录，则删除这个 pending bug 副本。
- 更新 `tests/alert-api.test.ts`：新增已处理不重建待处理副本、同 key 已处理时删除待处理副本、删除重复已处理幽灵记录三条回归测试。
